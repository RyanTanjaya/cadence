Project 5 — Web App (Full-Stack)

Habit Tracker

A full-stack web habit tracker built around proven gamification psychology — completion rings, flexible streaks with user agency, and competence feedback instead of empty badges.

React + Node.js PostgreSQL + Prisma Gestalt Closure Rings Research-backed Design

🎯 What It Does

Most habit apps feel like obligation factories — streaks that punish you, badges that mean nothing, and feature overload that buries the actual habit. This app is built differently, using the same psychological mechanisms behind Apple Watch's activity rings and Duolingo's streak system, applied with the restraint the research actually recommends.

The user creates habits, sets a realistic target frequency (not always "every day"), and each day sees a set of completion rings — one per habit. Tapping a ring fills it. When all rings close, the day is done. That's the primary loop.

⭕
Completion Rings
Each habit is a circular progress ring. Tapping fills it. The Gestalt principle of closure makes a 90% filled ring feel impossible to leave open — the brain is hardwired to want to close it. This is the same mechanic Apple Watch used to drive a 49.5% behavior change in 160,000 people.
🎯
Flexible Streaks
Users set their own target: "every day" or "4 out of 7 days". One built-in grace day per week so missing one day doesn't reset everything. Research shows uncontrollable streaks shift from motivational to obligational after ~31 days — this design avoids that trap.
📈
Competence Feedback
Real growth metrics: completion rate %, longest streak ever, personal records auto-celebrated. "Your yoga completion went from 43% to 71% this month" means something. Empty points and badges do not.
🗓️
12-Week Heatmap
GitHub-style contribution heatmap per habit showing patterns across 12 weeks. Makes consistency visible and satisfying without any external comparison or competition.
🎁
Weekly Reveal Card
Every Sunday, an animated summary card appears showing the week's performance — total completions, best day, any personal records. The user doesn't know what the score will be until they open the app: variable reward magnitude in action.
🔔
Smart Reminders
Per-habit notification at a user-set time. Automatically skipped if the habit is already completed for the day — no nagging for things already done.

🧠 Gamification Design (Research-Backed)

Every design decision here maps directly to one of Tim Gabe's 7 research-backed patterns from studying 500+ gamified apps. The decisions are documented so you can explain them in your case study and Upwork profile.

Core Philosophy
Most apps build the scoreboard and forget to build the game. PBL — Points, Badges, Leaderboards are the three most documented gamification failures in product history (LinkedIn, Foursquare, Google News all scrapped them). This app runs on completion drive (Apple Watch rings), streaks with user agency (Duolingo's model), and competence feedback (Peloton's personal records). Max 3 gamification mechanics — the S-curve research shows more than that reverses engagement.

What We're Using

✅ Pattern 6 — Completion Drive
Rings (Gestalt Closure)
The Gestalt principle of closure: a 90% filled circle creates an open loop the brain is hardwired to complete. Apple Watch drove 49.5% behavior change in 160,000 people with this single mechanic. Every habit = a ring that wants to be closed.
✅ Pattern 4 (Modified) — Streaks
Flexible Streaks with Agency
Duolingo runs streaks with user-chosen goal levels and freeze mechanics. Research: uncontrollable streaks cause FOMO and obligation after ~31 days. Here: user sets their own frequency target + 1 grace day/week. The streak is something they control, not something that controls them.
✅ Pattern 7 — Competence
Real Progress Metrics
Peloton members who see real output data (watts, personal records) work out 15% more than those who don't. Chess.com ELO. Garmin's training readiness. Show evidence of skill and consistency growth, not app-open frequency. Personal records auto-celebrated when set.
✅ Pattern 5 — Variable Reward
Weekly Reveal
Anticipation of an unknown reward size is a stronger engagement driver than a known one. Every Sunday, an animated reveal shows the week's summary. The user doesn't know their "score" until the card opens. Pull toward what's next, not fear of losing what's built.

What We're Avoiding

❌ Pattern 1 — PBL Fallacy
Generic Points & Badges
LinkedIn retired top voice badges in 2024 — badge-motivated users produced quantity over quality. Foursquare and Google News scrapped their systems for the same reason. No XP counter, no badge collection. Any milestone shown must represent real behavior.
❌ Pattern 3 — S-Curve
Feature Overload
Habitica (the most gamified habit app ever) had 100% of study participants experience counterproductive effects — managing the game layer buried the actual habit. This app ships 3 gamification mechanics max. MVP ships 2: rings + streaks.
❌ Pattern 4 (Rigid)
Unbreakable Streaks
Streaks the user can't influence become obligation, not motivation. The Nevada AG filed against Snapchat for addictive streak mechanics in 2024. No "you broke your streak" punishment screen. Grace days are built into the system by design, not as a premium feature.
❌ Pattern 2 (Unwinnable)
Global Leaderboards
Strava's insight: global leaderboards are unwinnable at scale and therefore demotivating. The only meaningful competition is against your own past self. All comparisons in this app are personal — this week vs. last week, this month vs. last month.

🗄️ Database Schema (Prisma)

// schema.prisma

model User {
  id           String    @id @default(cuid())
  email        String    @unique
  passwordHash String
  name         String
  createdAt    DateTime  @default(now())
  habits       Habit[]
}

model Habit {
  id          String       @id @default(cuid())
  userId      String
  user        User         @relation(fields: [userId], references: [id])
  name        String
  emoji       String       @default("⭐")
  color       String       @default("#8b5cf6")
  frequency   String       @default("daily")
  // 'daily' | 'weekly:4' (4x per week) | 'weekdays:1,3,5' (Mon/Wed/Fri)
  graceDays   Int          @default(1)
  sortOrder   Int          @default(0)
  isArchived  Boolean      @default(false)
  createdAt   DateTime     @default(now())
  completions Completion[]
}

model Completion {
  id        String   @id @default(cuid())
  habitId   String
  habit     Habit    @relation(fields: [habitId], references: [id], onDelete: Cascade)
  date      String   // 'YYYY-MM-DD'
  createdAt DateTime @default(now())

  @@unique([habitId, date])  // one completion per habit per day
}

Key Calculations (Pure TypeScript in src/utils/streakLogic.ts)

FunctionLogic
getCurrentStreak()Walk backwards from today through completion records. For 'daily': streak breaks when a day is missed AND the 7-day window's grace day is used. For 'weekly:N': check Mon–Sun windows — each needs N completions to count. Stop when a window fails.
getLongestStreak()Same logic, walk entire history, track the max consecutive count seen.
getCompletionRate(days)Count completed days in last N days ÷ expected days based on frequency × 100.
getWeeklySummary(week)Sum completions Mon–Sun. Find best day (most completions). Compare streaks to personal bests to detect new records.
ℹ️
Keep all streak logic as pure functions in src/utils/streakLogic.ts — pass in arrays of completion records, no DB calls inside. The backend calls these after fetching completions from Prisma. This makes streak logic easy to test in isolation.

KEY STREAK LOGIC (src/utils/streakLogic.ts — PURE functions, no DB calls)
===========================================================================
getCurrentStreak(frequency, graceDays, completionDates: string[]):
  - 'daily': walk backwards day by day. Each 7-day window allows graceDays free misses. Streak breaks when a day is missed AND that window's grace is exhausted.
  - 'weekly:N': walk backwards by Mon–Sun windows. Each window needs N completions. Streak = consecutive windows meeting target.
  - 'weekdays:1,3,5': only target days count. Walk backwards through target days only. 1 grace per 7-day window.

getLongestStreak(): same logic, walk entire history, track max.
getCompletionRate(days=30): completed_days / expected_days × 100.

All functions are PURE — accept string[] of 'YYYY-MM-DD' dates, no Prisma calls inside. Test in isolation before connecting to routes.

CONSTRAINTS & NOTES
===================
- All dates stored as 'YYYY-MM-DD' strings (never raw Date objects or UTC timestamps for date-only data).
- "Today" = client's local date (format with date-fns format(new Date(), 'yyyy-MM-dd')), sent as a string to all API calls. Server never infers today's date from server time.
- Completion toggle: POST to create, DELETE to remove. Use TanStack Query optimistic updates for instant ring fill feel.
- @@unique([habitId, date]) on Completion — Prisma's upsert handles concurrent taps gracefully.
- Weekly reveal: check on DashboardPage mount. If today is Monday and localStorage['weekRevealSeen_YYYY-MM-DD'] is not set (where date = this week's Monday), fetch last week's stats, show banner, set localStorage key on dismiss.
- JWT: store in localStorage. Axios interceptor reads it and sets Authorization: Bearer header on every request. On 401, redirect to /login and clear localStorage.
- graceDays applies per 7-day rolling window, not per calendar week.
- IDs: Prisma cuid() for all PKs.
- shadcn/ui components: Dialog for AddHabit, Table for stats, Tooltip for heatmap cells.
